## 一、功能概述

### 1.1 产品背景

Claude Code Runner 当前通过 Web 界面和 API 提供服务，但缺乏用户认证机制。任何人都可以访问系统并执行任务，存在安全风险。随着项目功能不断丰富（任务调度、文件浏览器等），需要对用户访问进行权限控制，支持多用户场景下的使用统计和资源配额管理。

### 1.2 功能定位

用户认证系统是 Phase 1: 安全与认证 阶段的核心功能，为 Claude Code Runner 提供完整的用户身份验证和访问控制能力。

### 1.3 核心价值

| 价值点 | 描述 |
|--------|------|
| 安全保障 | 防止未授权访问，保护系统资源 |
| 多用户支持 | 支持多个用户独立使用，各用户数据隔离 |
| API 访问 | 支持程序化访问，支持 API Key 认证 |
| 审计追溯 | 记录用户行为，支持安全审计 |

## 二、用户场景分析

### 2.1 目标用户

| 用户类型 | 使用场景 | 需求特点 |
|----------|----------|----------|
| 个人开发者 | 本地使用，无需多用户 | 简单认证，快速上手 |
| 团队管理员 | 管理团队成员，分配权限 | 用户管理、权限控制 |
| API 开发者 | 程序调用，执行自动化任务 | API Key 认证、密钥管理 |
| 运维人员 | 监控系统，查看日志 | 角色区分、审计追溯 |

### 2.2 用户故事

#### 故事 1：个人用户注册登录

作为个人用户，我希望能够注册账号并登录系统，这样我可以使用任务调度等高级功能。

**验收条件**：
- 用户可以使用邮箱/用户名注册账号
- 用户可以使用密码登录系统
- 登录后可以访问个人中心查看账号信息

#### 故事 2：API 开发者使用密钥访问

作为 API 开发者，我希望能够创建和管理 API 密钥，这样我可以通过程序调用系统功能。

**验收条件**：
- 用户可以创建 API 密钥
- 用户可以使用 API 密钥通过 `X-API-Key` Header 访问 API
- 用户可以撤销不再使用的 API 密钥

#### 故事 3：管理员管理用户

作为系统管理员，我希望能够管理用户账号，这样我可以禁用异常账号。

**验收条件**：
- 管理员可以查看所有用户列表
- 管理员可以禁用/启用用户账号
- 管理员可以重置用户密码

## 四、API 设计需求

### 4.1 认证接口

#### 4.1.1 用户注册

```
POST /api/auth/register

Request:
{
  "username": "user@example.com",  // 必填，邮箱格式
  "password": "secure_password",   // 必填，最小8位，包含字母和数字
  "name": "John Doe"              // 必填，显示名称
}

Response: 201 Created
{
  "user_id": "uuid",
  "username": "user@example.com",
  "name": "John Doe",
  "created_at": "2026-03-10T00:00:00Z"
}
```

#### 4.1.2 用户登录

```
POST /api/auth/login

Request:
{
  "username": "user@example.com",
  "password": "secure_password"
}

Response: 200 OK
{
  "access_token": "eyJhbGciOiJIUzI1NiIs...",
  "token_type": "bearer",
  "expires_in": 3600,
  "refresh_token": "refresh_token_string"
}
```

#### 4.1.3 获取当前用户

```
GET /api/auth/me

Headers: Authorization: Bearer <token>

Response: 200 OK
{
  "user_id": "uuid",
  "username": "user@example.com",
  "name": "John Doe",
  "is_active": true,
  "created_at": "2026-03-10T00:00:00Z"
}
```

#### 4.1.4 修改密码

```
PUT /api/auth/password

Headers: Authorization: Bearer <token>

Request:
{
  "old_password": "old_password",
  "new_password": "new_password"
}

Response: 200 OK
{
  "success": true,
  "message": "密码修改成功"
}
```

#### 4.1.5 刷新 Token

```
POST /api/auth/refresh

Request:
{
  "refresh_token": "refresh_token_string"
}

Response: 200 OK
{
  "access_token": "new_access_token",
  "token_type": "bearer",
  "expires_in": 3600
}
```

#### 4.1.6 登录限流说明

| 限流规则 | 值 |
|----------|-----|
| 限制范围 | 同一 IP 地址 |
| 时间窗口 | 5 分钟 |
| 最大尝试次数 | 10 次 |
| 超过限制后 | 返回 429 Too Many Requests |

```
Response: 429 Too Many Requests
{
  "success": false,
  "error": "登录尝试过于频繁，请 5 分钟后再试",
  "retry_after": 300
}
```

### 4.2 API 密钥接口

#### 4.2.1 创建 API 密钥

```
POST /api/keys

Headers: Authorization: Bearer <token>

Request:
{
  "name": "My API Key",        // 必填，密钥名称
  "expires_in": 86400          // 可选，过期时间（秒），默认无限
}

Response: 201 Created
{
  "key_id": "uuid",
  "name": "My API Key",
  "key": "sk-ccr-xxxxxxxxxxxxx",  // ⚠️ 仅在此次响应中返回，之后不再显示
  "created_at": "2026-03-10T00:00:00Z",
  "expires_at": "2026-03-11T00:00:00Z",
  "is_active": true
}
```

#### 4.2.2 列出 API 密钥

```
GET /api/keys

Headers: Authorization: Bearer <token>

Response: 200 OK
{
  "keys": [
    {
      "key_id": "uuid",
      "name": "My API Key",
      "prefix": "sk-ccr-xxx...",
      "created_at": "2026-03-10T00:00:00Z",
      "last_used": "2026-03-10T12:00:00Z",
      "expires_at": "2026-03-11T00:00:00Z",
      "is_active": true
    }
  ]
}
```

#### 4.2.3 撤销 API 密钥

```
DELETE /api/keys/{key_id}

Headers: Authorization: Bearer <token>

Response: 204 No Content
```

## 五、安全要求

### 5.1 密码安全

| 要求 | 描述 |
|------|------|
| 最小长度 | 密码最小 8 位 |
| 复杂度 | 必须包含字母和数字 |
| 存储 | 使用 bcrypt 哈希存储，禁止明文 |
| 传输 | 所有认证接口必须使用 HTTPS |

### 5.2 Token 安全

| 要求 | 描述 |
|------|------|
| 算法 | JWT 使用 HS256 算法 |
| 有效期 | 默认 1 小时，可配置 |
| 存储 | Token 存储在内存中（生产环境建议使用 Redis） |
| 刷新 | 支持 Refresh Token 刷新 |

### 5.3 API 密钥安全

| 要求 | 描述 |
|------|------|
| 密钥格式 | sk-ccr-{32位随机字符串}，总长度 39 字符 |
| 一次性显示 | 创建密钥时只返回一次，之后不再显示 |
| 哈希存储 | 密钥存储使用哈希值 |
| 前缀显示 | 列表接口只显示前缀（如 sk-ccr-abc1） |
| 过期设置 | 支持设置密钥过期时间 |

### 5.4 权限控制

| 要求 | 描述 |
|------|------|
| 密钥所属权 | API 密钥只能由创建者查看和管理 |
| 管理员权限 | 管理员功能需要特定角色标识 |
| 审计日志 | 记录用户登录、密钥创建等敏感操作 |

## 七、验收标准

### 7.1 功能验收

| 验收项 | 验收条件 |
|--------|----------|
| 用户注册 | 使用有效邮箱和符合复杂度要求的密码可以成功注册 |
| 用户登录 | 使用正确凭证可以登录并获取 Token |
| Token 访问 | 携带有效 Token 可以访问受保护资源 |
| API Key 访问 | 使用 X-API-Key Header 可以访问受保护资源 |
| 创建密钥 | 可以成功创建 API 密钥，且密钥只显示一次 |
| 撤销密钥 | 可以成功撤销 API 密钥，撤销后密钥立即失效 |
| 修改密码 | 用户可以成功修改自己的密码 |
| 密钥列表 | 列表接口不显示完整密钥，只显示前缀 |

### 7.2 安全验收

| 验收项 | 验收条件 |
|--------|----------|
| 密码存储 | 数据库中密码以哈希形式存储，无法还原 |
| Token 验证 | 过期 Token 无法访问受保护资源 |
| 密钥独立性 | 用户无法查看或操作他人的 API 密钥 |
| 权限控制 | 未认证用户无法访问受保护资源 |
| 注入防护 | SQL 注入、XSS 等常见攻击无效 |
| 用户禁用 Token 失效 | 用户被禁用后，所有 Token 立即失效 |

### 7.3 性能验收

| 验收项 | 验收条件 |
|--------|----------|
| 登录性能 | 登录接口响应时间 < 200ms |
| 认证性能 | Token 验证时间 < 10ms |
| 并发性能 | 100 并发请求下认证正常 |

## 九、与现有系统集成

### 9.1 认证中间件设计

认证系统通过 FastAPI 依赖注入方式集成到现有 API 体系中。

```
┌─────────────────────────────────────────────────────────┐
│                    FastAPI 应用                          │
├─────────────────────────────────────────────────────────┤
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐   │
│  │ Auth Middle │ → │ 业务逻辑     │ → │ 响应返回    │   │
│  │ (Depends)   │   │             │   │             │   │
│  └─────────────┘   └─────────────┘   └─────────────┘   │
└─────────────────────────────────────────────────────────┘
```

认证中间件支持两种认证方式（按优先级）：
1. **API Key 认证** - 通过 `X-API-Key` Header 传递
2. **JWT Bearer Token** - 通过 `Authorization: Bearer` Header 传递

### 9.2 现有 API 改造计划

| 现有 API | 改造前 | 改造后 | 优先级 |
|----------|--------|--------|--------|
| POST /api/task | 无需认证 | 需要认证 (P0) | P0 |
| GET /api/sessions | 无需认证 | 需要认证 (P0) | P0 |
| POST /api/scheduler/tasks | 无需认证 | 需要认证 (P0) | P0 |
| GET /api/scheduler/status | 无需认证 | 需要认证 (P1) | P1 |
| GET /api/files/tree | 无需认证 | 需要认证 (P1) | P1 |

### 9.3 认证流程

```
客户端请求
    ↓
检查 X-API-Key Header
    ↓ [存在]
验证 API Key
    ↓ [不存在]
检查 Authorization Header
    ↓ [Bearer Token]
验证 JWT Token
    ↓ [无效]
返回 401 Unauthorized
    ↓ [有效]
提取 user_id
    ↓
执行业务逻辑
    ↓
返回响应
```

### 9.4 用户禁用后 Token 处理

- 用户被禁用后，所有该用户的 JWT Token 和 API Key 立即失效
- 每次认证时检查用户 `is_active` 状态，非活跃用户返回 403 Forbidden
- Token 黑名单机制：禁用用户时将其有效 Token 加入黑名单

## 十、技术实现建议

### 10.1 技术选型

| 组件 | 技术方案 | 说明 |
|------|----------|------|
| 密码哈希 | bcrypt | 安全可靠 |
| JWT | python-jose | 支持 HS256 |
| 用户存储 | SQLite/JSON 文件 | 复用现有存储方式 |
| Token 存储 | 内存 | 生产环境可迁移到 Redis |

### 10.2 数据模型建议

```python
# 用户模型
class User:
    user_id: str          # UUID
    username: str        # 邮箱格式
    password_hash: str   # bcrypt 哈希
    name: str            # 显示名称
    is_active: bool      # 账号状态
    is_admin: bool       # 管理员标识
    created_at: datetime
    last_login: datetime

# API 密钥模型
class APIKey:
    key_id: str          # UUID
    user_id: str         # 所属用户
    key_hash: str        # 密钥哈希
    prefix: str          # 密钥前缀
    name: str            # 密钥名称
    created_at: datetime
    last_used: datetime
    expires_at: datetime
    is_active: bool
```

## 十、风险与对策

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| Token 泄露风险 | 高 | 实现 Token 刷新机制，支持用户撤销 Token |
| 密码暴力破解 | 中 | 实现登录次数限制，异常 IP 封禁 |
| API 密钥泄露 | 中 | 密钥只显示一次，提供密钥撤销功能 |
| 多用户数据隔离 | 高 | 严格的用户 ID 关联，所有操作需验证所属权 |
