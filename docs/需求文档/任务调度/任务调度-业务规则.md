# 任务调度-业务规则

## 1. 任务创建 (BR-001)

**业务规则**: 用户可以通过 API 或 Web 界面创建任务

| 规则编号 | 规则描述 | 优先级 | 实现状态 |
|---------|---------|--------|----------|
| BR-001-1 | 任务必须包含唯一标识符 `id`，使用 UUID 格式 | 高 | ✅ |
| BR-001-2 | 任务必须包含 `prompt` 字段，描述任务内容 | 高 | ✅ |
| BR-001-3 | 任务可指定 `workspace` 工作目录，默认为项目根目录 | 中 | ✅ |
| BR-001-4 | 任务可设置 `timeout` 超时时间，默认 600000ms（10分钟） | 中 | ✅ |
| BR-001-5 | 任务可设置 `auto_approve` 自动批准标志，默认 false | 高 | ✅ |
| BR-001-6 | 任务可限制 `allowed_tools` 可用工具列表 | 中 | ✅ |
| BR-001-7 | **【待实现】** 任务可指定 `cli_type` 选择执行的 CLI 工具 | 中 | ❌ |
| BR-001-8 | Task 模型从存储恢复时过滤未知字段，防止数据混合错误 | 高 | ✅ |

**任务创建方式**:
```
方式一: REST API
  POST /api/scheduler/tasks
  {
    "prompt": "任务描述",
    "workspace": "工作目录", 
    "timeout": 600000,
    "auto_approve": false,
    "allowed_tools": ["Read", "Write", "Edit"]
  }

方式二: Web 界面
  通过任务调度器界面创建任务
```

## 2. 任务队列管理 (BR-002)

**业务规则**: 任务按先进先出（FIFO）顺序执行

| 规则编号 | 规则描述 | 优先级 | 实现状态 |
|---------|---------|--------|----------|
| BR-002-1 | 任务存储在 `data/queue.json` 文件的 `tasks` 数组中 | 高 | ✅ |
| BR-002-2 | Cron 触发时，将任务放入 `data/queue.json` 队尾 | 高 | ✅ |
| BR-002-3 | 队列处理循环检查 `data/running.json` 是否为空 | 高 | ✅ |
| BR-002-4 | running.json 为空时，从 queue.json 取第一个任务执行 | 高 | ✅ |
| BR-002-5 | running.json 不为空时，等待当前任务执行完成 | 高 | ✅ |
| BR-002-6 | 同一时间 running.json 中最多只有一个运行中的任务 | 高 | ✅ |
| BR-002-7 | APScheduler 配置 `max_instances=1`, `coalesce=False` | 高 | ✅ |

## 3. 任务状态流转 (BR-003)

**业务规则**: 任务在整个生命周期中有明确的状态流转

```
状态流转图:

  ┌──────────────┐
  │   创建任务    │
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐      ┌──────────────┐
  │  queue.json  │◀────│  重试(次数<N) │
  │   (待执行)    │      └──────────────┘
  └──────┬───────┘              ▲
         │                      │
         ▼                      │
  ┌──────────────┐              │
  │running.json  │              │
  │   (执行中)    │              │
  └──────┬───────┘              │
         │                      │
    ┌────┴────┬─────────┐       │
    ▼         ▼         ▼       │
┌────────┐ ┌────────┐ ┌────────┐
│  成功   │ │  失败   │ │  取消   │
└────┬───┘ └────┬───┘ └────┬───┘
     │          │            │
     ▼          ▼            ▼
┌────────┐ ┌────────┐  ┌────────┐
│completed│ │ failed │  │cancelled│
│ .json   │ │ .json  │  │ .json   │
└─────────┘ └────────┘  └────────┘
```

| 状态 | 文件位置 | 说明 |
|------|---------|------|
| 待执行 | queue.json | 任务已创建，等待调度 |
| 执行中 | running.json | 任务正在执行 |
| 已完成 | completed.json | 任务执行成功 |
| 已失败 | failed.json | 任务超过最大重试次数后失败 |
| 已取消 | failed.json | 用户主动取消运行中的任务 |

**取消任务业务规则 (BR-003-CANCEL)**:
| 规则编号 | 规则描述 | 优先级 |
|---------|---------|--------|
| BR-003-C1 | 只有运行中(RUNNING)状态的任务可以被取消 | 高 |
| BR-003-C2 | 取消操作需要用户二次确认 | 高 |
| BR-003-C3 | 取消后任务状态变为 CANCELLED | 高 |
| BR-003-C4 | 取消后的任务不触发重试机制 | 高 |
| BR-003-C5 | 取消后的任务移动到 failed.json 存储 | 中 |

## 4. Cron 表达式解析 (BR-004)

**业务规则**: 系统支持标准 Cron 表达式定义定时任务

| 规则编号 | 规则描述 | 优先级 | 实现状态 |
|---------|---------|--------|----------|
| BR-004-1 | 支持 5 位格式: `分钟 小时 日 月 星期` | 高 | ✅ |
| BR-004-2 | 支持 6  位格式: `秒 分钟 小时 日 月 星期` | 中 | ✅ |
| BR-004-3 | 支持通配符 `*` 表示任意值 | 高 | ✅ |
| BR-004-4 | 支持间隔表达式 `*/n` 表示每 n 单位 | 高 | ✅ |
| BR-004-5 | 支持范围表达式 `a-b` 表示从 a 到 b | 中 | ✅ |
| BR-004-6 | 支持列表表达式 `a,b,c` 表示多个值 | 中 | ✅ |

**常用 Cron 表达式示例**:

| 表达式 | 说明 |
|--------|------|
| `0 9 * * *` | 每天 9:00 |
| `*/20 * * * *` | 每 20 分钟 |
| `0 */2 * * *` | 每 2 小时整点 |
| `0 9 * * 1-5` | 周一到周五 9:00 |
| `0 0 1 * *` | 每月 1 日 0:00 |

## 5. 定时任务触发机制 (BR-005)

**业务规则**: 定时任务到期时自动加入执行队列

| 规则编号 | 规则描述 | 优先级 | 实现状态 |
|---------|---------|--------|----------|
| BR-005-1 | 系统每 10 秒检查一次定时任务是否到期 | 高 | ✅ |
| BR-005-2 | 使用 `last_run` 记录上次执行时间防止重复执行 | 高 | ✅ |
| BR-005-3 | 到期任务创建副本加入 `queue.json`，原任务保留 | 高 | ✅ |
| BR-005-4 | 定时任务副本标记 `scheduled: true` 和 `scheduled_id` | 中 | ✅ |
| BR-005-5 | 跳过 `enabled: false` 的禁用任务 | 高 | ✅ |
| BR-005-6 | 无效 Cron 表达式的任务不执行并记录错误 | 中 | ✅ |

**定时任务配置结构**:
```json
{
  "id": "daily-report",
  "name": "每日报告",
  "cron": "0 9 * * *",
  "prompt": "任务描述",
  "workspace": "工作目录",
  "timeout": 1800000,
  "auto_approve": true,
  "allowed_tools": null,
  "enabled": true,
  "last_run": "2026-02-25T01:00:00.000Z",
  "next_run": "2026-02-26 09:00:00"
}
```

## 6. 防止重复执行 (BR-006)

**业务规则**: 使用 `next_run` 检查定时任务是否到期

| 规则编号 | 规则描述 | 实现状态 |
|---------|---------|----------|
| BR-006-1 | `next_run` 为 `null` 时，服务启动时触发执行并将任务加入队列，同时计算并设置新的 `next_run` | ✅ |
| BR-006-2 | `next_run` 不为 `null` 且 `next_run <= 当前时间` 时，触发执行 | ✅ |
| BR-006-3 | 首次创建时 `next_run` 可为 `null`，启用时应计算并设置 | ✅ |

**触发执行流程**:
1. 调度器启动时加载所有已启用的任务
2. 对每个任务检查触发条件：
   - 条件A: `next_run` 存在且已过期 (`next_run <= 当前时间`)
   - 条件B: `next_run` 为 `null` 或空字符串（适用于服务运行期间新增的任务）
3. 满足任一条件时：
   - 将任务加入执行队列 (`queue.json`)
   - 使用 Cron 表达式计算下一次执行时间
   - 更新 `scheduled.json` 中的 `next_run` 字段

## 9. 任务重试机制 (BR-009)

**业务规则**: 任务失败后自动重试，超过次数后标记为最终失败

| 规则编号 | 规则描述 | 优先级 | 实现状态 |
|---------|---------|--------|----------|
| BR-009-1 | 默认最大重试次数为 2 次（共执行 3 次） | 高 | ✅ |
| BR-009-2 | 每次重试更新任务的 `retries` 计数 | 高 | ✅ |
| BR-009-3 | 重试任务保留在队列头部等待下次轮询 | 中 | ✅ |
| BR-009-4 | 超过最大重试次数后移动到 `failed.json` | 高 | ✅ |

**重试逻辑**:
```
任务失败 → retries = (retries || 0) + 1
         ↓
   retries < max_retries (2)?
    ├─ 是 → 保留在队列，下次重试
    └─ 否 → 移动到 failed.json
```

## 13. 重复任务ID处理 (BR-013)

**业务规则**: 系统允许创建相同ID但不同name的定时任务，通过自动编号避免冲突

| 规则编号 | 规则描述 | 优先级 | 实现状态 |
|---------|---------|--------|----------|
| BR-013-1 | ID+name 完全相同时更新现有任务 | 高 | ✅ |
| BR-013-2 | ID 相同但 name 不同时，自动给 name 添加数字编号后缀 | 高 | ✅ |
| BR-013-3 | `get()` 方法优先返回已启用的任务 | 高 | ✅ |
| BR-013-4 | `get_enabled()` 自动去重，相同ID只保留一个 | 高 | ✅ |
| BR-013-5 | 调度器启动时检查过期任务并立即执行 | 高 | ✅ |
| BR-013-6 | 过期任务执行后自动更新 `next_run` 为下一次执行时间 | 高 | ✅ |

**编号规则示例**:
```
1. 创建: ID=task-1, name="产品总监任务" 
   → 保存为: "产品总监任务"

2. 再次创建: ID=task-1, name="产品总监任务"
   → 更新现有任务

3. 创建: ID=task-1, name="产品总监任务(v8)"
   → 保存为: "产品总监任务(v8)(1)"

4. 创建: ID=task-1, name="产品总监任务(v9)"
   → 保存为: "产品总监任务(v9)(2)"
```

**查询优先级**:
- `get(task_id)`: 优先返回 `enabled=true` 的任务，如果没有则返回第一个匹配的任务
- `get_enabled()`: 按ID去重，相同ID只保留一个
