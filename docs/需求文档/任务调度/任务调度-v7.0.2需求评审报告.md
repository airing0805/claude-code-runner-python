## 第2次迭代评审详情

### 修改内容

#### 1. 需求规格文档修改 (任务调度-需求规格.md)

**新增需求**:
- FR-028.1: 用户可以取消运行中的任务
  - 取消后任务状态变为 CANCELLED
  - 取消操作需要用户二次确认
  - 取消后的任务不触发重试机制
  - 取消后的任务移动到 failed.json 文件

- FR-027 增强停止调度器行为描述:
  - 优雅停止：等待当前运行任务完成后停止调度器
  - 强制停止：立即停止调度器，当前运行任务继续执行但不再启动新任务

**安全需求增强 (NFR-006~NFR-011)**:
- NFR-006: 补充工作目录安全验证的详细规则
  - 禁止路径遍历：不允许使用 `..` 跳出项目根目录
  - 禁止访问系统目录：不允许访问 `/etc`, `/root`, `C:\Windows` 等系统敏感目录
  - 白名单机制：仅允许访问配置文件中定义的允许目录
  - 路径规范化：统一处理相对路径和绝对路径，防止绕过验证
- NFR-010: API 访问安全（当前版本为单机应用，无认证要求）
  - 未来扩展：支持 API Token 认证
  - 未来扩展：支持 IP 白名单访问控制
- NFR-011: 敏感数据保护
  - 日志中不应包含 API Key、密码等敏感信息
  - 响应中不应返回敏感配置信息
  - 环境变量敏感信息需要脱敏处理

**测试覆盖率验证明确 (NFR-020)**:
- 使用 pytest-cov 工具验证覆盖率
- 在 CI/CD 流程中自动执行覆盖率检查
- 核心模块（scheduler, storage, executor）覆盖率要求 ≥ 90%

**NFR 编号修正**: 重新编号为 NFR-001 ~ NFR-022，消除编号冲突

#### 2. 业务规则文档修改 (任务调度-业务规则.md)

**状态流转图更新**:
```
新增 CANCELLED 状态分支：
  RUNNING → CANCELLED → failed.json
```

**新增取消任务业务规则 (BR-003-CANCEL)**:
| 规则编号 | 规则描述 | 优先级 |
|---------|---------|--------|
| BR-003-C1 | 只有运行中(RUNNING)状态的任务可以被取消 | 高 |
| BR-003-C2 | 取消操作需要用户二次确认 | 高 |
| BR-003-C3 | 取消后任务状态变为 CANCELLED | 高 |
| BR-003-C4 | 取消后的任务不触发重试机制 | 高 |
| BR-003-C5 | 取消后的任务移动到 failed.json 存储 | 中 |

#### 3. 用户故事文档修改 (任务调度-用户故事.md)

**新增用户故事**:

**US-009: 取消运行中任务**
- 验收标准:
  - 可以取消运行中(RUNNING)状态的任务
  - 取消操作需要用户二次确认
  - 取消后任务状态变为 CANCELLED
  - 取消后的任务不触发重试机制
  - 取消后的任务显示在失败任务列表中
- 优先级: 高
- 关联需求: FR-028.1

**US-010: 处理任务执行异常**
- 验收标准:
  - 超时任务自动终止并标记为失败
  - 资源耗尽时有适当的错误处理
  - CLI工具崩溃时能够捕获并记录错误
  - 所有异常情况都有详细的日志记录
  - 异常后系统能够继续正常运行
- 优先级: 高
- 关联需求: BR-010, BR-011, NFR-019

**新增验收测试场景**:
- 场景6: 开发工程师取消运行中的任务
- 场景7: DevOps工程师处理任务异常

**优先级矩阵更新**:
- 高优先级用户故事新增 US-009, US-010
