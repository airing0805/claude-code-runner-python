## 1. 需求概述

### 1.1 模块目标

任务队列管理模块是任务调度系统的核心组成部分，负责管理待执行任务的排队、调度和执行顺序。该模块允许用户手动添加任务到队列、查看队列状态、删除特定任务或清空整个队列。

### 1.2 设计原则

> **核心设计原则**（重要）
> - **队列积压是正常的**：同一个定时任务在队列中多次出现是正常现象。Cron 每次触发都会添加任务到队列，如果任务执行慢，队列会积压，这是正常的计划安排。
> - **禁止去重**：不应该基于 prompt 进行去重。不同的定时任务可能有相同的 prompt，但它们是独立的任务，不应该互相干扰。
> - **队列反映真实状态**：队列应该真实反映任务状态，用户可以看到有多少任务在等待。如果去除重复，用户就看不到任务堆积，无法判断任务执行是否有问题。

## 2. 功能需求分析

### 2.1 FR-001: 添加任务到队列

#### 需求描述
用户可以提交任务描述（prompt）到任务队列

#### 用户场景

| 场景 | 描述 |
|------|------|
| 场景1 | 开发人员需要批量执行代码审查任务 |
| 场景2 | DevOps工程师需要安排自动化测试任务 |
| 场景3 | 用户需要定时执行重复性AI任务 |

#### 输入

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| prompt | string | 是 | 任务描述/提示词，最大长度无限制 |
| workspace | string | 否 | 工作目录，默认为当前目录 "." |
| timeout | integer | 否 | 超时时间（秒），范围1-3600，默认600秒（10分钟） |
| auto_approve | boolean | 否 | 是否启用自动批准模式，默认false |
| allowed_tools | array[string] | 否 | 允许使用的工具列表 |

#### 输出

| 字段 | 类型 | 说明 |
|------|------|------|
| success | boolean | 操作是否成功 |
| data | object | 返回的任务信息 |
| message | string | 操作消息 |

#### 业务规则

1. prompt 不能为空或仅包含空白字符
2. workspace 必须在允许的目录列表中（安全验证）
3. timeout 必须在 1-3600 秒范围内
4. allowed_tools 必须是预定义工具白名单的子集
5. 任务添加成功后返回任务ID（UUID格式）

#### 验收标准

- [ ] 可以成功添加任务到队列
- [ ] 不合法的 prompt 被拒绝
- [ ] 超时时间超出范围时被拒绝
- [ ] 工作目录不在白名单时被拒绝
- [ ] 工具列表包含非白名单工具时被拒绝
- [ ] 返回正确的任务ID

### 2.2 FR-002: 指定工作目录

#### 需求描述
支持指定工作目录（workspace），默认为当前工作目录

#### 用户场景

| 场景 | 描述 |
|------|------|
| 场景1 | 用户需要在特定项目目录下执行任务 |
| 场景2 | 用户需要为不同项目分别执行任务 |

#### 输入

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| workspace | string | 否 | "." | 工作目录路径 |

#### 业务规则

1. 支持相对路径和绝对路径
2. 必须进行安全验证（路径遍历攻击防护）
3. 必须在允许访问的目录白名单中
4. 如果不提供，默认为 "."（当前工作目录）

#### 安全规则

- 禁止使用 `..` 跳出项目根目录
- 禁止访问系统目录（如 `/etc`, `/root`, `C:\Windows` 等）
- 仅允许访问配置文件中定义的允许目录

#### 验收标准

- [ ] 可以指定相对路径
- [ ] 可以指定绝对路径
- [ ] 路径遍历攻击被阻止
- [ ] 系统目录访问被拒绝
- [ ] 默认值正确生效

### 2.3 FR-003: 设置任务超时时间

#### 需求描述
支持设置任务超时时间（1秒-1小时，默认10分钟）

#### 用户场景

| 场景 | 描述 |
|------|------|
| 场景1 | 简单任务需要较短超时时间 |
| 场景2 | 复杂重构任务需要较长超时时间 |

#### 输入

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| timeout | integer | 否 | 600 | 超时时间（秒），范围1-3600 |

#### 业务规则

1. 最小值为 1 秒
2. 最大值为 3600 秒（1小时）
3. 默认值为 600 秒（10分钟）
4. 超过时间限制任务被强制终止

#### 验收标准

- [ ] 可以设置1秒超时
- [ ] 可以设置3600秒超时
- [ ] 默认超时为600秒
- [ ] 超出范围的值被拒绝
- [ ] 超时后任务被正确终止

### 2.4 FR-004: 配置允许使用的工具

#### 需求描述
支持配置允许使用的工具列表

#### 用户场景

| 场景 | 描述 |
|------|------|
| 场景1 | 只读任务只需要 Read、Glob、Grep 工具 |
| 场景2 | 完整开发任务需要所有工具 |

#### 输入

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| allowed_tools | array[string] | 否 | null（全部允许） | 允许使用的工具列表 |

#### 工具白名单

| 工具名称 | 功能描述 |
|----------|----------|
| Read | 读取文件内容 |
| Write | 写入文件内容 |
| Edit | 编辑文件内容 |
| Glob | 文件路径匹配 |
| Grep | 文本搜索 |
| Bash | 执行shell命令 |
| Task | 创建子任务 |
| TodoWrite | 写入TODO列表 |
| WebFetch | 获取网页内容 |
| WebSearch | 网络搜索 |
| NotebookEdit | Jupyter notebook编辑 |

#### 业务规则

1. 如果不提供 allowed_tools，则所有工具都允许使用
2. 如果提供，必须是白名单工具的子集
3. 任务执行时只能使用列表中的工具

#### 验收标准

- [ ] 不提供时所有工具可用
- [ ] 可以限制为部分工具
- [ ] 非白名单工具被拒绝
- [ ] 执行时只能使用允许的工具

### 2.5 FR-005: 启用自动批准模式

#### 需求描述
支持启用自动批准模式，跳过工具操作确认

#### 用户场景

| 场景 | 描述 |
|------|------|
| 场景1 | 无人值守的自动化任务 |

#### 输入

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| auto_approve | boolean | 否 | false | 是否启用自动批准模式 |

#### 业务规则

1. 默认值为 false（需要手动批准）
2. 设置为 true 时，所有工具操作自动批准
3. 此选项应谨慎使用，可能导致意外文件修改

#### 验收标准

- [ ] 默认需要确认
- [ ] 启用后无需确认
- [ ] 可以随时切换模式

## 4. 数据模型

### 4.1 Task 数据结构

```python
@dataclass
class Task:
    id: str                          # 任务唯一标识 (UUID)
    prompt: str                      # 任务描述/提示词
    workspace: str = "."             # 工作目录
    timeout: int = 600               # 超时时间（秒）
    auto_approve: bool = False      # 是否自动批准工具操作
    allowed_tools: Optional[list[str]] = None  # 允许使用的工具列表
    created_at: str = field(...)     # 创建时间 (ISO 8601)
    started_at: Optional[str] = None # 开始执行时间
    finished_at: Optional[str] = None  # 结束时间
    retries: int = 0                # 当前重试次数
    status: TaskStatus = field(...) # 任务状态
    source: TaskSource = field(...) # 任务来源
    scheduled_id: Optional[str] = None   # 定时任务ID
    scheduled_name: Optional[str] = None  # 定时任务名称
    result: Optional[dict] = None    # 执行结果
    error: Optional[str] = None      # 错误信息
    files_changed: list[str] = field(default_factory=list)  # 变更的文件
    tools_used: list[str] = field(default_factory=list)     # 使用过的工具
    cost_usd: Optional[float] = None # 消耗费用（美元）
    duration_ms: Optional[int] = None  # 执行耗时（毫秒）
```

### 4.2 枚举类型

#### TaskStatus

| 状态值 | 说明 |
|--------|------|
| pending | 待执行 |
| running | 运行中 |
| completed | 已完成 |
| failed | 失败 |
| cancelled | 已取消 |

#### TaskSource

| 来源值 | 说明 |
|--------|------|
| manual | 手动添加 |
| scheduled | 定时任务自动触发 |
| immediate | 立即执行定时任务 |

## 5. API 接口规范

### 5.1 添加任务到队列

```
POST /api/scheduler/tasks
```

**请求体：**
```json
{
  "prompt": "帮我审查代码",
  "workspace": ".",
  "timeout": 600,
  "auto_approve": false,
  "allowed_tools": ["Read", "Write", "Edit", "Glob", "Grep"]
}
```

**响应：**
```json
{
  "success": true,
  "data": {
    "id": "uuid-xxxx-xxxx",
    "prompt": "帮我审查代码",
    "workspace": ".",
    "timeout": 600,
    "status": "pending",
    "created_at": "2026-03-07T10:00:00Z"
  },
  "message": "任务已添加到队列"
}
```

### 5.2 获取队列列表

```
GET /api/scheduler/tasks
```

**响应：**
```json
{
  "success": true,
  "data": {
    "tasks": [...],
    "total": 5
  },
  "message": ""
}
```

### 5.3 删除队列任务

```
DELETE /api/scheduler/tasks/{task_id}
```

**响应：**
```json
{
  "success": true,
  "data": null,
  "message": "任务已从队列中删除"
}
```

### 5.4 清空队列

```
DELETE /api/scheduler/tasks/clear
```

**响应：**
```json
{
  "success": true,
  "data": null,
  "message": "队列已清空"
}
```
