# Claude Code Runner 与 OpenClaw 竞品对比分析

本文档从多个维度对 Claude Code Runner 和 OpenClaw 进行对比分析。

---

## 1. 项目定位与愿景

| 维度 | Claude Code Runner | OpenClaw |
|------|-------------------|----------|
| **核心定位** | Claude Code 的 Web API 封装 | 多渠道个人 AI 助手网关 |
| **解决的问题** | 让 Claude Code 可通过 HTTP API 调用 | 让 AI 助手通过已有消息渠道对话 |
| **目标用户** | 开发者、需要编程任务自动化的团队 | 普通用户、需要多渠道 AI 助手 |
| **愿景** | 提供可编程的 AI 编程能力 | 本地化、始终在线的个人 AI 助手 |

### 关键差异
- **Claude Code Runner**：面向开发者，是工具类产品
- **OpenClaw**：面向普通用户，是完整产品

## 11. 多项目/多工作区对比

### Agent 架构差异

| 维度 | Claude Code Runner | OpenClaw |
|------|-------------------|----------|
| **主代理概念** | ❌ 无 | ✅ Gateway 主代理 |
| **子代理创建** | 通过 prompt 指令 | 通过 sessions_spawn 工具 |
| **子代理管理** | AgentManager | subagent-registry |
| **Workspace** | Task 级 | Agent 级 |
| **多代理配置** | 每次 Task 指定 | 配置文件预定义 |

### 项目管理对比

| 维度 | Claude Code Runner | OpenClaw |
|------|-------------------|----------|
| **项目管理** | 基础 | 完整 |
| **实现概念** | working_dir | Agent + Workspace |
| **多代理** | 通过 prompt 灵活创建 | 配置文件 + sessions_spawn |
| **工作区隔离** | 文件系统 | 完整作用域 |
| **消息路由** | ❌ | ✅ 多渠道路由 |
| **资源配额** | ❌ | ✅ 可配置 |
| **会话管理** | 基础 | 完整 |
| **配置覆盖** | 基础 | ✅ 模型/行为覆盖 |
| **跨 Workspace** | ❌ | ✅ 通过 agentId 参数 |

### 架构示意

**Claude Code Runner** (无主代理):
```
Task (指定 workspace + prompt)
    │
    ├── 主执行 Agent (Task 创建)
    │
    └── 子代理 (通过 prompt 指令创建)
         - "创建一个子代理来处理 X"
         - 每个子代理有独立状态跟踪
```

**OpenClaw** (有 Gateway 主代理):
```
Gateway (主代理，处理所有消息)
    │
    ├── 路由到预配置的 Agent
    │
    └── sessions_spawn 工具 → Subagent
         - runtime: "subagent" | "acp"
         - 每个 Subagent 独立工作区
```

### Claude Code Runner 当前能力

```
- 每个任务可指定不同 working_dir
- 每个会话独立工作目录
- 通过 prompt 指令创建子代理
- AgentManager 统一管理子代理
- 无主代理概念，Task 即是入口
```

### OpenClaw 当前能力

```
- Gateway 主代理：统一入口
- 多 Agent 实例：通过配置预定义
- Workspace 隔离的工作目录
- 消息自动路由到对应 Agent
- sessions_spawn 工具创建 Subagent
- 每个 Agent/Subagent 可配置不同模型
- 完整的会话生命周期管理
- subagent-depth 限制子代理深度
```

### 结论

- **Claude Code Runner**：适合灵活的任务执行场景，无主代理概念，prompt 即控制
- **OpenClaw**：适合多项目、需要隔离和路由的复杂场景，有明确的 Gateway 主代理

## 13. 系统级 vs 项目级：核心定位差异

> 本章分析两个产品的核心定位差异，帮助选择合适的产品。

### 定位对比

| 维度 | Claude Code Runner | OpenClaw |
|------|-------------------|----------|
| **层级** | 项目级 | 系统级 |
| **任务性质** | 持续项目，离散任务 | 主要是离散任务 |
| **协同方式** | 流程式（评审流转） | 对话式（消息触发） |
| **输出** | 文档 + 代码 | 对话响应 |

### OpenClaw - 系统级多Agent协同

**定位**：多渠道个人 AI 助手网关

**适用场景**：
- 个人 AI 助手（通过 WhatsApp、Telegram 对话）
- 消息驱动的自动化
- 短任务执行（查天气、设提醒）
- 多渠道消息聚合

**不适用场景**：
- 持续迭代的项目开发
- 多阶段评审流程
- 长期维护的代码库
- 完整的文档+代码产出

### Claude Code Runner - 项目级Agent协同

**定位**：项目开发任务执行引擎

**适用场景**：
- 持续迭代的项目开发
- 多阶段评审流程（产品规划→技术设计→开发→审核）
- 长期维护的代码库
- 文档+代码完整产出
- 定时任务驱动的项目迭代

**不适用场景**：
- 简单的对话式 AI 助手
- 多渠道消息集成

### 评审机制对比

| 机制 | OpenClaw | Claude Code Runner |
|------|----------|-------------------|
| **评审定义** | 需要通过 sessions_spawn + prompt 实现 | ✅ 内置评审流程 |
| **修订循环** | 手动处理 | ✅ 自动触发修订 |
| **状态追踪** | 无专门管理 | ✅ 任务状态管理 |
| **角色调用** | 通过 prompt | ✅ 专门的 Skills (/产品总监, /产品经理) |

### 互补关系

两者并非完全替代，而是**互补**：

```
┌─────────────────────────────────────────┐
│           用户交互层                      │
│   (消息渠道 / API / Web 界面)           │
└───────────────┬─────────────────────────┘
                │
    ┌───────────┴───────────┐
    ↓                       ↓
OpenClaw              Claude Code Runner
系统级                 项目级
    │                       │
消息处理               任务执行
对话交互               评审流转
    │                       │
    └───────────┬───────────┘
                ↓
        可整合使用
        (OpenClaw 触发 CCR 任务)
```

**整合示例**：
1. OpenClaw 接收用户消息："实现用户登录功能"
2. 通过 Cron 或 Slash 命令触发 Claude Code Runner
3. Claude Code Runner 执行完整的产品规划→技术设计→开发流程
4. 结果返回 OpenClaw 通过消息渠道通知用户
