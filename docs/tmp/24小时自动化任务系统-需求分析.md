## 一、产品概述

### 1.1 产品定位

Claude Code 24小时自动化任务系统是一个基于 PM2 进程管理器的 **任务驱动型自动化执行平台**，旨在让 AI CLI 工具（如 Claude Code、Qoder CLI）能够 24/7 不间断地执行预设任务。

### 1.2 目标用户

| 用户群体 | 使用场景 | 核心需求 |
|---------|---------|---------|
| 开发者 | 自动化代码审查、测试运行 | 批量任务处理、无人值守执行 |
| 项目经理 | 定时生成报告、监控项目状态 | 定时任务调度、结果通知 |
| 运维人员 | 系统维护、日志分析 | 高可用性、故障自动恢复 |
| AI Agent 集成者 | 多 Agent 协同工作 | 多 CLI 工具支持、任务编排 |

### 1.3 核心价值主张

1. **高可用性**: PM2 守护进程确保服务永不中断
2. **自动化处理**: 任务队列机制支持异步、批量任务处理
3. **故障自愈**: 进程崩溃自动重启，任务失败自动重试
4. **可扩展性**: 插件化架构支持多种 CLI 工具

## 二、功能模块需求

### 2.1 任务管理模块

#### BR-001: 任务创建

**业务规则**: 用户可以通过多种方式创建任务

| 规则编号 | 规则描述 | 优先级 |
|---------|---------|--------|
| BR-001-1 | 任务必须包含唯一标识符 `id`，格式建议为 `task-{timestamp}` | 高 |
| BR-001-2 | 任务必须包含 `prompt` 字段，描述任务内容 | 高 |
| BR-001-3 | 任务可指定 `workspace` 工作目录，默认为项目根目录 | 中 |
| BR-001-4 | 任务可设置 `timeout` 超时时间，默认 600000ms（10分钟） | 中 |
| BR-001-5 | 任务可设置 `autoApprove` 自动批准标志，默认 false | 高 |
| BR-001-6 | 任务可限制 `allowedTools` 可用工具列表 | 中 |
| BR-001-7 | 任务可指定 `cliType` 选择执行的 CLI 工具 | 中 |

**任务创建方式**:
```
方式一: Windows 批处理脚本
  scripts\add-task.bat "任务描述" -w 工作目录 -t 超时时间 -y

方式二: Node.js CLI
  node src/add-task.js "任务描述" --workspace 路径 --timeout 时间

方式三: 直接编辑 JSON 文件
  编辑 tasks/queue.json
```

#### BR-002: 任务队列管理

**业务规则**: 任务按先进先出（FIFO）顺序执行

| 规则编号 | 规则描述 | 优先级 |
|---------|---------|--------|
| BR-002-1 | 任务存储在 `tasks/queue.json` 文件的 `list` 数组中 | 高 |
| BR-002-2 | 系统每次从队列头部取出一个任务执行 | 高 |
| BR-002-3 | 任务执行期间队列中其他任务等待 | 高 |
| BR-002-4 | 同一时间只能执行一个任务（串行执行） | 高 |

#### BR-003: 任务状态流转

**业务规则**: 任务在整个生命周期中有明确的状态流转

```
状态流转图:

  ┌──────────────┐
  │   创建任务    │
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐      ┌──────────────┐
  │  queue.json  │◀────│  重试(次数<N) │
  │   (待执行)    │      └──────────────┘
  └──────┬───────┘              ▲
         │                      │
         ▼                      │
  ┌──────────────┐              │
  │running.json  │              │
  │   (执行中)    │              │
  └──────┬───────┘              │
         │                      │
    ┌────┴────┐                 │
    ▼         ▼                 │
┌────────┐ ┌────────┐           │
│  成功   │ │  失败   │───────────┘
└────┬───┘ └────┬───┘
     │          │ (次数>=N)
     ▼          ▼
┌────────┐ ┌────────┐
│completed│ │ failed │
│ .json   │ │ .json  │
└─────────┘ └────────┘
```

| 状态 | 文件位置 | 说明 |
|------|---------|------|
| 待执行 | queue.json | 任务已创建，等待调度 |
| 执行中 | running.json | 任务正在执行 |
| 已完成 | completed.json | 任务执行成功 |
| 已失败 | failed.json | 任务超过最大重试次数后失败 |

### 2.2 定时任务模块

#### BR-004: Cron 表达式解析

**业务规则**: 系统支持标准 Cron 表达式定义定时任务

| 规则编号 | 规则描述 | 优先级 |
|---------|---------|--------|
| BR-004-1 | 支持 5 位格式: `分钟 小时 日 月 星期` | 高 |
| BR-004-2 | 支持 6 位格式: `秒 分钟 小时 日 月 星期` | 中 |
| BR-004-3 | 支持通配符 `*` 表示任意值 | 高 |
| BR-004-4 | 支持间隔表达式 `*/n` 表示每 n 单位 | 高 |
| BR-004-5 | 支持范围表达式 `a-b` 表示从 a 到 b | 中 |
| BR-004-6 | 支持列表表达式 `a,b,c` 表示多个值 | 中 |

**常用 Cron 表达式示例**:

| 表达式 | 说明 |
|--------|------|
| `0 9 * * *` | 每天 9:00 |
| `*/20 * * * *` | 每 20 分钟 |
| `0 */2 * * *` | 每 2 小时整点 |
| `0 9 * * 1-5` | 周一到周五 9:00 |
| `0 0 1 * *` | 每月 1 日 0:00 |

#### BR-005: 定时任务触发机制

**业务规则**: 定时任务到期时自动加入执行队列

| 规则编号 | 规则描述 | 优先级 |
|---------|---------|--------|
| BR-005-1 | 系统每 10 秒检查一次定时任务是否到期 | 高 |
| BR-005-2 | 使用 `lastRun` 记录上次执行时间防止重复执行 | 高 |
| BR-005-3 | 到期任务创建副本加入 `queue.json`，原任务保留 | 高 |
| BR-005-4 | 定时任务副本标记 `scheduled: true` 和 `scheduledId` | 中 |
| BR-005-5 | 跳过 `enabled: false` 的禁用任务 | 高 |
| BR-005-6 | 无效 Cron 表达式的任务不执行并记录错误 | 中 |

**定时任务配置结构**:
```json
{
  "tasks": [
    {
      "id": "daily-report",
      "name": "每日报告",
      "cron": "0 9 * * *",
      "prompt": "任务描述",
      "workspace": "工作目录",
      "timeout": 1800000,
      "autoApprove": true,
      "allowedTools": null,
      "enabled": true,
      "lastRun": "2026-02-25T01:00:00.000Z",
      "nextRun": "2026-02-26 09:00:00"
    }
  ]
}
```

#### BR-006: 防止重复执行

**业务规则**: 使用分钟级时间戳比较防止同一任务重复触发

| 规则编号 | 规则描述 |
|---------|---------|
| BR-006-1 | 将 `lastRun` 和当前时间截断到分钟级别比较 |
| BR-006-2 | 计算 `lastRun` 之后的下一个执行时间 `nextAfterLastRun` |
| BR-006-3 | 如果 `nextAfterLastRun <= 当前时间` 且不等于 `lastRun`，则触发执行 |

### 2.3 任务执行模块

#### BR-007: CLI 插件管理

**业务规则**: 系统通过插件架构支持多种 CLI 工具

| 规则编号 | 规则描述 | 优先级 |
|---------|---------|--------|
| BR-007-1 | 每个 CLI 工具作为独立插件配置 | 高 |
| BR-007-2 | 插件配置定义命令行参数映射规则 | 高 |
| BR-007-3 | 支持通过 `stdin` 或命令行参数传递任务参数 | 中 |
| BR-007-4 | 未指定 `cliType` 时使用默认 CLI（claude） | 高 |

**支持的 CLI 类型**:

| CLI 类型 | 命令 | 自动批准参数 | 工具限制参数 |
|---------|------|------------|------------|
| claude | `claude` | `--dangerously-skip-permissions` | `--allowedTools` |
| qoder | `qodercli` | `--yolo` | `--allowed-tools` |

**CLI 配置结构**:
```json
{
  "defaultCliType": "claude",
  "cliTypes": {
    "claude": {
      "command": "claude",
      "parameters": {
        "prompt": { "method": "stdin", "required": true },
        "autoApprove": { "flag": "--dangerously-skip-permissions" }
      }
    },
    "qoder": {
      "command": "qodercli",
      "parameters": {
        "prompt": { "flag": "-p", "required": true },
        "autoApprove": { "flag": "--yolo" }
      }
    }
  }
}
```

#### BR-008: 任务执行流程

**业务规则**: 任务执行遵循严格的流程控制

| 规则编号 | 规则描述 | 优先级 |
|---------|---------|--------|
| BR-008-1 | 执行前删除环境变量 `CLAUDECODE` 以允许嵌套会话 | 高 |
| BR-008-2 | 任务信息写入 `running.json` 标记执行状态 | 高 |
| BR-008-3 | 通过 `stdin` 传递任务描述给 CLI | 高 |
| BR-008-4 | 捕获并记录 `stdout` 和 `stderr` 输出 | 中 |
| BR-008-5 | 执行完成后清空 `running.json` | 高 |

**执行环境处理**:
```javascript
// Windows
spawnCommand = 'cmd.exe';
spawnArgs = ['/c', 'set CLAUDECODE= && claude --参数'];

// Unix/Linux/Mac
spawnCommand = 'env';
spawnArgs = ['-u', 'CLAUDECODE', 'claude', '--参数'];
```

### 2.4 错误处理与重试模块

#### BR-009: 任务重试机制

**业务规则**: 任务失败后自动重试，超过次数后标记为最终失败

| 规则编号 | 规则描述 | 优先级 |
|---------|---------|--------|
| BR-009-1 | 默认最大重试次数为 2 次（共执行 3 次） | 高 |
| BR-009-2 | 每次重试更新任务的 `retries` 计数 | 高 |
| BR-009-3 | 重试任务保留在队列头部等待下次轮询 | 中 |
| BR-009-4 | 超过最大重试次数后移动到 `failed.json` | 高 |

**重试逻辑**:
```
任务失败 → retries = (retries || 0) + 1
         ↓
   retries < maxRetries (2)?
    ├─ 是 → 保留在队列，下次重试
    └─ 否 → 移动到 failed.json
```

#### BR-010: 超时处理

**业务规则**: 单个任务执行时间超过限制时强制终止

| 规则编号 | 规则描述 | 优先级 |
|---------|---------|--------|
| BR-010-1 | 默认超时时间为 600000ms（10 分钟） | 高 |
| BR-010-2 | 超时后发送 `SIGTERM` 信号终止进程 | 高 |
| BR-010-3 | 超时视为任务失败，触发重试机制 | 高 |
| BR-010-4 | 任务级别可自定义超时时间 | 中 |

#### BR-011: 进程异常处理

**业务规则**: 捕获并处理子进程的各种异常情况

| 异常类型 | 处理方式 | 错误信息 |
|---------|---------|---------|
| spawn 错误 | 记录日志，触发重试 | `任务执行失败: {err.message}` |
| 信号终止 | 视为失败 | `Process killed by signal: {signal}` |
| 非零退出码 | 视为失败 | `Process exited with code: {code}` |

### 2.6 安全模块

#### BR-014: 权限控制

**业务规则**: 通过多层级控制限制任务执行权限

| 规则编号 | 规则描述 | 风险等级 |

| BR-014-3 | `allowedTools` 限制任务可使用的工具集 | 中 |
| BR-014-4 | `workspace` 限制任务的工作目录 | 中 |

**业务规则**: 建议使用环境隔离降低风险

| 规则编号 | 规则描述 |
|---------|---------|
| BR-015-1 | 测试环境和生产环境使用不同的任务队列 |
| BR-015-2 | 敏感操作不使用 `autoApprove: true` |
| BR-015-3 | 定期审查 `completed.json` 了解执行的操作 |
| BR-015-4 | 重要数据定期备份 |

## 三、数据结构规范

### 3.1 任务对象 (Task)

```typescript
interface Task {
  // 必填字段
  id: string;           // 任务唯一标识，建议格式: task-{timestamp}
  prompt: string;       // 任务描述/指令

  // 可选字段
  workspace?: string;   // 工作目录，默认项目根目录
  timeout?: number;     // 超时时间(ms)，默认 600000
  autoApprove?: boolean;// 自动批准，默认 false
  allowedTools?: string[] | null; // 可用工具列表，null 表示不限制
  cliType?: string;     // CLI 类型，默认 'claude'
  createdAt?: string;   // 创建时间，ISO 8601 格式

  // 定时任务字段
  scheduled?: boolean;  // 是否为定时任务触发
  scheduledId?: string; // 对应的定时任务 ID

  // 执行状态字段
  retries?: number;     // 已重试次数
  startedAt?: string;   // 开始执行时间
  completedAt?: string; // 完成时间
  failedAt?: string;    // 失败时间
  status?: 'success' | 'failed'; // 执行状态
  error?: string;       // 错误信息
}
```

### 3.2 定时任务对象 (ScheduledTask)

```typescript
interface ScheduledTask {
  // 必填字段
  id: string;           // 定时任务唯一标识
  cron: string;         // Cron 表达式
  prompt: string;       // 任务描述

  // 可选字段
  name?: string;        // 任务名称
  workspace?: string;   // 工作目录
  timeout?: number;     // 超时时间
  autoApprove?: boolean;// 自动批准
  allowedTools?: string[] | null; // 可用工具列表
  enabled?: boolean;    // 是否启用，默认 true
  createdAt?: string;   // 创建时间

  // 状态字段
  lastRun?: string;     // 上次执行时间
  nextRun?: string;     // 下次预计执行时间
}
```

### 3.3 文件结构

```
tasks/
├── queue.json       # 待执行任务队列 { "list": [Task] }
├── running.json     # 当前运行任务 Task | {}
├── completed.json   # 已完成任务 { "list": [Task] }
├── failed.json      # 失败任务 { "list": [Task] }
└── scheduled.json   # 定时任务配置 { "tasks": [ScheduledTask] }
```
