# Python 任务调度包对比分析

## 当前项目状态

### 当前使用的包
**无** - 当前项目完全自己实现了调度逻辑，依赖 `asyncio` 和自定义的定时循环。

### 当前实现方式
- 使用 `asyncio.create_task()` 和 `asyncio.wait_for()` 实现异步任务执行
- 自定义轮询循环（每 10 秒检查一次）
- 使用 JSON 文件存储任务状态
- 自定义 Cron 表达式解析器

## 可用的 Python 任务调度包

### 1. APScheduler (Advanced Python Scheduler)

**特点**：
- 纯 Python 实现，无外部依赖
- 支持多种调度器（Cron、Interval、Date）
- 内置持久化存储（Memory、SQLAlchemy、MongoDB、Redis）
- 支持异步执行
- 支持任务失败重试
- 内置并发控制

**优点**：
- 成熟稳定，使用广泛
- 文档完善
- 支持异步执行
- 内置 Cron 支持，无需自己实现

**缺点**：
- 不支持进程隔离（需要自己实现）
- 依赖进程池管理

**适用场景**：
- 中小型任务调度
- 不需要分布式部署

**示例代码**：
```python
from apscheduler.schedulers.asyncio import AsyncIOScheduler
from apscheduler.triggers.cron import CronTrigger

scheduler = AsyncIOScheduler()

# 添加定时任务
scheduler.add_job(
    func=execute_task,
    trigger=CronTrigger.from_crontab("*/20 * * * *"),
    id="my-task",
    max_instances=1,  # 限制并发数
)

# 启动
scheduler.start()
```

### 2. Celery

**特点**：
- 分布式任务队列系统
- 支持消息代理（RabbitMQ、Redis）
- 支持任务结果存储
- 支持任务链和组
- 内置失败重试和任务监控
- 支持定时任务（Celery Beat）

**优点**：
- 完全的分布式支持
- 成熟稳定，生产级质量
- 丰富的监控和管理工具（Flower）
- 支持任务优先级和速率限制

**缺点**：
- 需要额外部署消息代理（Redis/RabbitMQ）
- 学习曲线较陡峭
- 对于单机场景可能过度设计

**适用场景**：
- 大规模分布式任务调度
- 需要高可用和负载均衡

### 3. RQ (Redis Queue)

**特点**：
- 基于 Redis 的简单任务队列
- 轻量级，易于使用
- 支持任务重试
- 内置 Web 监控界面
- 支持任务失败通知

**优点**：
- 简单易用
- 依赖少（只需 Redis）
- 内置任务状态监控

**缺点**：
- 不支持复杂的 Cron 表达式
- 需要额外实现定时调度
- 不支持任务链

**适用场景**：
- 简单的异步任务队列
- 不需要复杂调度规则

### 4. Dramatiq

**特点**：
- 基于 Redis/RabbitMQ 的任务队列
- 支持 Celery 兼容的 API
- 内置定时任务支持
- 支持任务链和组
- 内置重试机制

**优点**：
- 比 Celery 更简单
- 内置定时任务支持
- 支持异步执行

**缺点**：
- 依赖 Redis 或 RabbitMQ
- 社区规模不如 Celery

**适用场景**：
- 需要 Celery 功能但希望更简单

### 5. schedule (轻量级)

**特点**：
- 极简 API
- 只支持固定间隔和固定时间
- 不支持 Cron 表达式
- 无持久化

**优点**：
- 极其简单
- 零依赖

**缺点**：
- 功能太简陋
- 不支持复杂调度规则
- 不适合生产环境

**适用场景**：
- 简单的定时任务
- 快速原型开发

### 6. Crontab (系统级)

**特点**：
- 使用系统 crontab
- 进程完全隔离
- 稳定性极高

**优点**：
- 系统级稳定
- 完全进程隔离
- 无需额外代码

**缺点**：
- 需要系统权限
- 不支持动态修改
- 任务状态需要自己管理

## 推荐方案对比

| 方案 | 复杂度 | 进程隔离 | 分布式 | 适用场景 |
|------|--------|---------|--------|---------|
| **保持现状** | 高 | 无 | 无 | 小型项目 |
| **APScheduler** | 中 | 无 | 无 | 中小型项目 |
| **APScheduler + subprocess** | 中 | 有 | 无 | 中小型项目（推荐） |
| **Celery** | 高 | 有 | 有 | 大型分布式 |
| **RQ + APScheduler** | 中 | 有 | 无 | 中小型项目 |
| **系统 crontab** | 低 | 有 | 无 | 简单定时任务 |

## 针对本项目的建议

### 推荐：APScheduler + subprocess 模式

**理由**：
1. APScheduler 提供成熟的 Cron 调度和任务管理
2. 使用 subprocess 确保进程隔离（与稳定版本一致）
3. 保留现有存储层（JSON 文件）
4. 迁移成本相对较低

**实现思路**：
```python
from apscheduler.schedulers.asyncio import AsyncIOScheduler
from apscheduler.triggers.cron import CronTrigger
import asyncio.subprocess

scheduler = AsyncIOScheduler()

async def execute_task_with_isolation(task):
    """使用进程隔离执行任务"""
    cmd = ["claude", "--dangerously-skip-permissions"]
    process = await asyncio.create_subprocess_exec(
        *cmd,
        stdin=asyncio.subprocess.PIPE,
        stdout=asyncio.subprocess.PIPE,
        cwd=task.workspace,
        env={**os.environ, "CLAUDECODE": ""}
    )
    stdout, stderr = await process.communicate(input=task.prompt.encode())
    return process.returncode == 0

scheduler.add_job(
    func=execute_task_with_isolation,
    trigger=CronTrigger.from_crontab(scheduled.cron),
    id=scheduled.id,
    max_instances=1,  # 确保同一任务不会并发执行
)

scheduler.start()
```

### 替代：Celery（如果需要分布式）

**理由**：
1. 完整的生产级任务队列
2. 内置任务状态管理
3. 支持任务优先级和速率限制
4. 有完善的监控工具

**缺点**：
1. 需要部署 Redis/RabbitMQ
2. 迁移成本较高
3. 对于单机场景可能过度设计

## 最终建议

**优先选择**：APScheduler + subprocess

**原因**：
1. 最小改动：保留现有存储层和 API
2. 进程隔离：使用 subprocess 实现与稳定版本一致的隔离
3. 成熟稳定：APScheduler 是广泛使用的成熟库
4. 学习成本低：API 简单直观

**如果 APScheduler 无法满足**：考虑升级到 Celery（需要额外基础设施）
