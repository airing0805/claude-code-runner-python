# 文档命名规范

> 本文件定义项目中各类文档的命名规范，旨在提高文档的可查找性和可维护性。

## 1. 概述

本文档规范适用于 `docs/` 目录下的所有文档，包括但不限于：
- 需求文档
- 技术设计文档
- 测试文档
- 规划文档

## 2. 目录结构

```
docs/
├── 需求文档/              # 需求规格说明
├── 技术设计/              # 技术实现方案
├── 测试/                  # 测试用例和计划
├── plan/                 # 规划类文档
│   ├── 功能规划/          # 功能版本规划
│   ├── 版本报告/          # 版本进度报告
│   ├── 官方功能/          # 官方功能分析
│   └── 其他/              # 其他规划
├── analysis/             # 分析类文档
├── samples/              # 示例数据
└── 文档命名规范.md        # 本文档
```

## 3. 文档分类

### 3.1 需求文档

记录功能的需求规格说明。

**命名格式**：`功能名-需求.md`

### 3.2 技术设计文档

记录功能的技术实现方案。

**命名格式**：`功能名-技术设计.md`

### 3.3 测试文档

记录功能的测试用例和测试计划。

**命名格式**：`功能名-测试.md`

## 5. 命名规则

### 5.1 字符限制

| 规则 | 说明 |
|------|------|
| 分隔符 | 使用 `-`（中划线）连接 |
| 空格 | 不允许使用空格 |
| 中文 | 使用中文描述功能名 |
| 特殊字符 | 仅允许使用 `-` 和中文 |

### 5.2 避免的命名

```markdown
# 错误示例
项目 管理.md        # ❌ 包含空格
项目管理.md         # ❌ 缺少后缀类型
ProjectManage.md   # ❌ 使用英文
项目管理_需求.md    # ❌ 使用下划线
```

```markdown
# 正确示例
当前会话-需求.md    # ✅ 使用中文 + 中划线 + 类型后缀
```

### 5.3 版本号标注

对于带版本号的功能需求，在功能名中包含版本号：

**示例**：
- `Agent监控-需求.md` （v0.3.6 版本）
- `技能系统管理-需求.md` （v0.3.5 版本）

版本号应与 `plan/功能规划/v0.x开发计划.md` 中的版本规划保持一致。




## 7. 关联文档命名

当某个文档是另一个文档的子功能或具体业务时，使用「功能-子功能」的命名方式：

### 7.1 子功能文档

如果一个功能包含多个子功能，且子功能可能独立查阅：

**命名格式**：`父功能-子功能.md`

**示例**：
```
需求文档/
├── 当前会话-任务输入.md    # 子功能：任务输入
├── 当前会话-输出展示.md    # 子功能：输出展示
└── 历史记录-会话列表.md    # 子功能：会话列表
```

### 7.2 引用关系

当文档 A 引用文档 B 时：
1. 在文档 A 中使用相对路径引用
2. 优先将公共部分抽取为独立文档

**示例**：

在 `当前会话-需求.md` 中引用 `历史记录-需求.md` 的会话列表功能：

```markdown
> 会话列表功能与 [历史记录-需求.md](./历史记录-需求.md) 完全相同，详见该文档。
```

## 8. 常见问题

> **说明**：以下为常见问题解答，仅供参考，非强制规范。

**Q: 一个功能既有需求文档又有技术设计文档，如何关联？**

A: 使用相同的命名前缀，通过后缀区分类型：
- `当前会话-需求.md`
- `当前会话-技术设计.md`
- `当前会话-测试.md`

**Q: 文档应该在哪个目录？**

A: 根据文档类型选择目录：
- 需求类 → `需求文档/`
- 实现类 → `技术设计/`
- 测试类 → `测试/`

**Q: 如何处理跨版本的文档？**

A: 版本号标注在功能名中，保持与开发计划版本一致。

**Q: 三个目录的文档必须完全对应吗？**

A: 是的，除了后缀外，三个目录的文档名应保持一致。如果某个功能在某类文档中不存在（如非菜单对应的独立文档），需要明确标注。
